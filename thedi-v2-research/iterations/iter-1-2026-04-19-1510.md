## Executive Summary

Three decisions are already made and drive everything below.

1. **Voice workflow = interview-bot + voice-note silence-fallback.** Ramesh answers 4–6 Socratic Q&A prompts by text; an LLM drafts prose anchored to his verbatim phrasings; a rubric-based critic gates the draft. When Ramesh is silent ≥2 weeks, the pipeline offers an on-device voice-memo fallback (MacWhisper local transcription — never raw audio over a wire). Pure few-shot drafting (outliner → drafter → critic from exemplars alone) is **disqualified**: the [Sep 2025 authorship-imitation paper (arXiv 2509.14543)](https://arxiv.org/html/2509.14543v1) caps blog-register style imitation at 19–65% accuracy, below the threshold that keeps Ramesh's Substack from reading as slop. See [Round 3 A](../round3/A-voice-workflow-decision.md).
2. **Ops model = Ramesh self-hosts his own Butterbase app.** Ramesh owns the account, the billing card, and every API key. Ashish ships a one-shot installer + `docs/ROTATION.md` and holds **no standing production credentials**. A 2-week Ashish-hosted staging bridge is permitted for voice calibration, then secrets rotate and Ashish's instance is deleted. This is the only configuration under which the "Ashish silent 4 weeks" test passes. See [Round 3 C](../round3/C-production-ops-model.md).
3. **North Star = conversation-with-Ramesh-required.** Ramesh's stated goal ("follower network; over time, paid subs") expands into three incompatible Year-1 builds: paid-primary, sponsorship-primary, advisory-funnel-primary. The honest Year-1 math under each ([Round 3 B](../round3/B-north-star-conversation.md), from [Brief 04](../round1/04-newsletter-economics.md)) is: ~$200–600/mo paid; ~$400–900/mo sponsor; or a single $15K advisory engagement that dwarfs either. Ashish cannot pick for Ramesh. Ashish's recommendation is (C) with (B) as free byproduct — but the build pauses on Ramesh's answer.

**Recommended architecture** (one paragraph). A small, explicit graph of Butterbase serverless functions with Postgres state: daily scout (reused from v1) → weekly topic-picker email to Ramesh → interview-bot Q&A session → drafter ([IonRouter kimi-k2.5](../round1/06-ionrouter-model-selection.md)) anchored to Q&A verbatim + extracted cadence markers → rubric-based critic ([gpt-oss-120b](../round1/06-ionrouter-model-selection.md), separate context, different model family) → conditional rewriter, max 2 rounds → HITL approval gate in Beehiiv CMS → manual paste to Substack ([no safe Substack write API exists](../round1/03-substack-beehiiv-apis.md)) → auto-syndicate LinkedIn + X + Substack Notes. Voice-note-first is the silence-fallback branch. Every IonRouter response asserts `model == expected_model_for_stage`; mismatch hard-fails and emails Ramesh.

**Phased roadmap** (one paragraph). Phase 1 (2 weeks, <90 min/task) unblocks the Ramesh conversation: send the North Star one-pager, propose the comp structure, ship the interview-bot MVP on Thedi v1's existing scout + Resend infrastructure. Phase 2 (weeks 3–6) builds the full graph — critic loop, installer, rotation runbook, silent-failure detection, model-eval harness. Phase 3 (post first subscriber cohort) adds analytics aligned to the chosen North Star, rubric-recalibration automation, and the founding-member tier if (A) or (B) was picked.

**Compensation recommendation.** MIT-licensed OSS on Ashish's personal GitHub + a written California Labor Code §2870 carve-out letter from Saviynt on file *before* the next commit + Ramesh self-hosts + no money flows + OSS-maintainer best-effort support (≤2 hrs/wk social norm). Retainer, gift-then-host, and rev-share all silently become unpaid contracting inside a reporting chain. See [Brief 07](../round1/07-power-dynamics-comp.md).

**The single biggest risk.** IonRouter silently routes to a different model mid-pipeline. Voice dilution is the one failure mode brief 02 says kills the Substack, and a model swap to a cheaper OSS variant for drafting would produce slop for weeks before anyone notices. The highest-leverage control in the entire system is a one-line `response.model == expected` assertion per call site with email-Ramesh-on-mismatch. If one line of code ships first, it is this one.

### Node rationales

**Scout (daily cron, reused from Thedi v1).** arxiv + HN with a scoring rubric tuned to agentic-AI / DevOps / SRE. No changes from v1. Brief 05 adds a recycle/sequel filter (critic report gap) — before proposing a topic, check Postgres `published_posts` for semantic overlap with the last 12 topics; suppress duplicates. Cost: ~30 min to add.

**Weekly topic-picker email (new).** Thursday 07:00 PT, Ramesh gets an email listing the top 3 candidate topics from the week's scout output; replying with a number (or clicking a link) selects. This is the first HITL gate. Replaces brief 05's implicit "Thedi picks and Ramesh approves later." Rationale: topic selection is the highest-leverage editorial decision; doing it first avoids 2–3 rounds of wasted LLM spend on topics Ramesh wouldn't have picked. Ramesh effort: ~2 minutes.

**Interview-bot (new, primary draft input).** Once Ramesh picks a topic, Thedi generates 4–6 Socratic Q&A prompts ("What's the failure mode you see most often with X?", "Tell me about a time Y failed"). Ramesh answers by text in an emailed form or `/#/q` web UI. This is the [Spiral v3](https://every.to/on-every/introducing-spiral-v3-an-ai-writing-partner-with-taste) pattern, adapted. Why this over exemplar-only drafting: [Brief 02](../round1/02-voice-preservation.md) cites [arXiv 2509.14543](https://arxiv.org/html/2509.14543v1) — blog-register style imitation from exemplars alone collapses to 19–65%; anchoring drafts to Ramesh's verbatim answers sidesteps that ceiling. Also handles cold-start (Ramesh may have no existing corpus). Rejected: Brief 01's outliner → drafter → critic pattern and Brief 05's prose-from-outline generation, both downstream of the unreliable few-shot voice mechanism.

**Drafter — IonRouter kimi-k2.5.** [Brief 06](../round1/06-ionrouter-model-selection.md) names Kimi K2.5 as the voice-critical choice: EQ-Bench Creative 1,663.8 (top of IonRouter's OSS pool, ~87% of Claude Sonnet 4.6's score at ~1/5 the cost per [evy.so benchmarks](https://evy.so/compare/best-llms-for-writing/)). Drafter inputs: Q&A verbatim + extracted cadence markers (from cold-start seed interview) + topic sources from scout. Drafter is explicitly instructed to preserve Ramesh's exact phrasings where possible.

**Critic — IonRouter gpt-oss-120b.** Separate call, separate context, different model family from drafter. Scores on: voice_fidelity (weighted 2×), factual_accuracy, concreteness, flow_coherence, slop_absence. `gpt-oss-120b` at $0.020/$0.095 per M is ~20× cheaper than Kimi — makes 2 rounds affordable. Critic outputs rubric scores + ≤5 specific edits. Rejected: same-model critic (collective delusion — [Nature npj AI 2025](https://www.nature.com/articles/s44387-025-00045-3)); critic with same context as drafter ([LangChain reflection agents](https://www.langchain.com/blog/reflection-agents) admits this fails without external evidence).

**Rewriter — same model as drafter, max 2 rounds.** [Brief 05](../round1/05-multi-agent-orchestration.md) cites [Self-Refine (NeurIPS 2023)](https://arxiv.org/abs/2303.17651) and [Reflexion](https://arxiv.org/abs/2303.11366): quality plateaus after rounds 2–3 on long-form prose. 2 rounds is the defensible default; 3rd round conditional on rubric score. Hard cap: 2.

**HITL gate #2 — Beehiiv CMS approval.** Beehiiv serves as the draft CMS because [Brief 03](../round1/03-substack-beehiiv-apis.md) confirms it has a real REST API with `POST /publications/{id}/posts` (status=draft), version history, and scheduled-publish primitives. Ramesh reviews/edits in Beehiiv's editor. Beehiiv's AUP bans "publications that rely entirely on AI-generated material without meaningful human input" ([Beehiiv AUP](https://www.beehiiv.com/aup)) — the interview-bot workflow is defensible because Ramesh's Q&A answers *are* the meaningful human input, resolving the Round 2 critic's contradiction #3.

**Publish — manual paste to Substack.** [Brief 03](../round1/03-substack-beehiiv-apis.md): Substack has no public publishing API in 2026; their TOS explicitly bans "processes activated while you are not logged in," making reverse-engineered session-cookie automation a ban risk on Ramesh's account. Manual paste is 5–10 min/post, well within the 2-hr/wk budget. Ramesh keeps Substack because his stated goal is a Substack (network effects, Notes, paid-subs UX) — [Brief 04](../round1/04-newsletter-economics.md) agrees even under the consulting-funnel North Star.

**Auto-syndicate.** LinkedIn via official [Posts API](https://learn.microsoft.com/en-us/linkedin/shared/api-guide/concepts/rate-limits) (`w_member_social` OAuth scope). X via pay-per-use ($0.01/post, negligible). Substack Notes manual because the reverse-API path is TOS-unsafe ([Brief 03](../round1/03-substack-beehiiv-apis.md)). LinkedIn Articles/Newsletters have no API, so they stay out of scope.

**Silence-fallback — voice-note-first.** When Ramesh is silent ≥2 weeks, the pipeline offers an alternate path: phone voice memo → **on-device MacWhisper** ([Jeff Triplett 2025](https://micro.webology.dev/2025/04/30/voice-dictation-with-ai-and/)) → transcript → same drafter/critic stack. Hard rule per [Round 3 A](../round3/A-voice-workflow-decision.md): no raw audio over any wire. No Whisper API, no Groq ASR. This protects against Ramesh being an IAM-company employee routing voice with incidental Saviynt context through a 4+ processor chain.

**Silent-failure detection (cross-cutting).** Per [Round 3 C](../round3/C-production-ops-model.md):
- **Model-ID assertion on every IonRouter call** — `response.model == expected`, hard-fail, email Ramesh. Highest-leverage single control. 10 LOC total.
- **Weekly health email to Ramesh (Mon 07:00 PT)** — posts drafted/approved/published, model-assertion pass rate, scout dedup stats, rubric-score trend, IonRouter spend WoW, key-expiry countdowns. If this email fails to send 2 weeks running → escalation to both Ramesh and Ashish.
- **Cron heartbeat watcher** — scout writes to `heartbeats` on every run; if no heartbeat in 26 hours, email Ramesh. Catches Butterbase-side failures.
- **Key-expiry watcher** — T-7 reminder, T-0 circuit breaker on 401/403.
- **Budget ceiling alert** — weekly IonRouter spend > $2 → email (baseline is <$0.20/wk).
- **Voice-drift detector** — if 3-post rolling mean `voice_fidelity` score drops >1 point vs. prior 8-post mean, flag in next health email. Cheap proxy for the full rubric-recalibration loop.

**Orchestrator — Butterbase function with explicit state machine** in Postgres (`outline_pending → qa_in_progress → drafting → critiquing → rewriting → review_pending → published`). No LangGraph, no CrewAI — both are Python-first and don't fit Butterbase's Deno serverless runtime ([Brief 01](../round1/01-sota-pipelines.md)). A 5-node state machine in plain SQL is tractable.

## Compensation / Arrangement Recommendation

*This section is load-bearing. Read it before writing any more code.*

### Recommended arrangement

1. **MIT-licensed open-source release** under Ashish's personal GitHub (`platformy` or personal handle), not Saviynt GitHub.
2. **Written §2870 carve-out letter from Saviynt Legal on file before the next commit.** Template filed alongside moonlighting disclosure. Budget 1 hour of a California employment attorney (~$400–600) to review before filing, per [Brief 07](../round1/07-power-dynamics-comp.md).
3. **Ramesh self-hosts his own Butterbase account.** He owns the app, the billing, the keys. Ashish has no standing credentials in production after the Phase 2 handoff.
4. **No money flows between coworkers.** No retainer, no flat fee, no rev-share, no 1099, no LLC.
5. **OSS-maintainer best-effort support from Ashish**, capped at the ≤2-hr/wk social norm. GitHub Issues is the documented channel. Ashish answers on his schedule, not Ramesh's.

### Why this and not each alternative

- **Retainer (D in [Brief 07](../round1/07-power-dynamics-comp.md))** — Ongoing monthly fee from Ramesh to Ashish creates a reporting-chain conflict of interest. Ashish's future manager is paying his employee. Every month becomes a potential dispute. Disqualified.
- **Gift-then-host (B+Option 2 from [Round 3 C](../round3/C-production-ops-model.md))** — Ashish builds free, Ashish hosts free, Ashish holds keys. This silently becomes unpaid contracting in a reporting chain. [Brief 07 §2 B analysis](../round1/07-power-dynamics-comp.md): asymmetric gifts to supervisors create implicit debt and ethical-policy violations. The "Ashish silent 4 weeks" test fails at the first expired key. Disqualified.
- **Revenue share (E)** — Ashish's income is a function of Ramesh's writing output. Creates a bizarre dynamic where Ashish's future manager is de-facto being compensated by his own employee's post cadence. Re-negotiation every quarter. Tax mess. Disqualified.
- **Flat fee + 1099 (C)** — Mechanically clean, but still a direct payment from coworker to coworker that needs Saviynt conflict-of-interest disclosure. Only the fallback if Ramesh refuses OSS.
- **LLC contractor (G)** — $800/yr CA franchise tax + admin overhead for a ≤2-hr/wk engagement. Only justified if the dollar amount is material and C is the chosen structure.
- **"We'll figure it out" (F)** — Universally disqualified across every career-advice, employment-law, and gift-economy source [Brief 07](../round1/07-power-dynamics-comp.md) cites. The failure mode, not an option.

### What the OSS framing explicitly supports

- Public repo, permissive license, Ramesh self-installs.
- Ashish's obligation is the obligation of any maintainer to any user: none beyond goodwill.
- Ashish's §2870 position maximally preserved — prior-art-by-Ashish, timestamped, public.

### What the OSS framing does NOT support (so Ashish must not drift into these)

- Ashish holding Ramesh's IonRouter/Beehiiv/Resend/LinkedIn/Anthropic keys.
- Ashish rotating those keys on a schedule.
- Ashish responding at 2am when Ramesh's subscriber digest didn't send.
- Ashish debugging a Beehiiv AUP flag on Ramesh's account.

All four are contracting. If any persist past the 2-week Phase 2 staging bridge, the arrangement has silently become Structure C and needs re-papering. [Round 3 C §6](../round3/C-production-ops-model.md) is explicit about this.

### The fallback if Ramesh rejects OSS

Structure C: flat fee, one invoice, written contract, frozen scope, Saviynt conflict-of-interest disclosure filed *before* the invoice. Not the recommendation; the only safe alternative if OSS is off the table.

### The one sentence for Ashish to say to Ramesh

> "I'll build it, I'll help you install it on your own Butterbase account, and from then on it's yours. I'll keep the repo healthy the same way any OSS maintainer does. I don't hold your keys and I'm not on call."

### What not to say

> "It's just a favor, don't worry about it."

That phrasing is the canonical setup to resentment and scope creep. [Brief 07 §5](../round1/07-power-dynamics-comp.md) names the word "just" as the canonical scope-creep tell in every freelance-contract failure-mode writeup.

## Appendix: 7 sub-question briefs

Each brief is summarized in one paragraph, with the decisions it drove. The full briefs are the appendix.

- [**Brief 01 — SOTA agentic content pipelines**](../round1/01-sota-pipelines.md). Surveyed 2025–2026 production agentic-writing patterns. Key finding: bounded, graph-structured workflows with typed state + one HITL gate ship; unbounded autonomous agents fail 41–87% of the time. Recommended few-shot voice anchoring as the drafter mechanism — which Brief 02 later contradicted, and which Round 3 A resolved by adopting interview-bot instead. Drove the graph-pipeline-in-Butterbase architecture and the rejection of LangGraph/CrewAI on a Deno runtime.

- [**Brief 02 — Voice preservation**](../round1/02-voice-preservation.md). The load-bearing brief. Cited [arXiv 2509.14543](https://arxiv.org/html/2509.14543v1) establishing that few-shot style imitation collapses to 19–65% accuracy in blog/forum register (vs. 95–97% in structured domains). Disqualified the outliner → drafter → critic pattern as the primary workflow. Proposed voice-note-first (Option A) and interview-bot (Option B) as alternatives. Drove the Round 3 A decision to adopt Option B primary + Option A silence-fallback.

- [**Brief 03 — Substack + Beehiiv APIs**](../round1/03-substack-beehiiv-apis.md). Confirmed Substack has no public publishing API and that reverse-engineered session-cookie automation violates TOS ("processes activated while not logged in"). Beehiiv has a real REST API and is the viable staging CMS, subject to its AUP clause banning "AI-generated material without meaningful human input" — which the interview-bot workflow satisfies. Drove: publish on Substack (manual paste), stage in Beehiiv, cross-post via LinkedIn Posts API + X pay-per-use. Ruled out LinkedIn Newsletters (no API) and HN submissions (no API, etiquette-forbidden).

- [**Brief 04 — Newsletter economics**](../round1/04-newsletter-economics.md). Stress-tested the "$1.2–2K/mo at 5K subs" estimate. Paid-subs alone nets $350–1,200/mo at Year-1 5K subs — implausible at the high end, plausible at the low. Sponsorships $400–900/mo realistic; one advisory engagement ($15K) dwarfs either. Drove the Round 3 B three-scenario North Star conversation; explicitly rejected "paid subs as Year-1 North Star" as misreading Ramesh's stated goal.

- [**Brief 05 — Multi-agent orchestration**](../round1/05-multi-agent-orchestration.md). Cited Stanford STORM + StoryWriter (CIKM 2025) for the research-scout → outliner → drafter decomposition pattern; cited Self-Refine + Reflexion for the "quality plateaus after rounds 2–3" rule. Google Research's 2025 scaling-agents paper sets the ceiling: >4 agent roles yields diminishing returns. Drove the 4-role (scout, drafter, critic, rewriter) + 2-round cap + 2-HITL-gate structure. Its outliner-as-drafter recommendation was overridden by Brief 02's voice evidence; the rest survives.

- [**Brief 06 — IonRouter model selection**](../round1/06-ionrouter-model-selection.md). Mapped each pipeline stage to an IonRouter model: qwen3.5-122b-a10b for research synthesis, glm-5 for outline, **kimi-k2.5 for drafting** (voice-critical), **gpt-oss-120b for critique**. Total cost ~$0.037/post. Drove the per-stage model choices in the architecture; also flagged the silent-model-swap risk that became the single biggest failure mode in Round 3 C (model-ID assertion circuit breaker).

- [**Brief 07 — Power dynamics and compensation**](../round1/07-power-dynamics-comp.md). Legal and sociological analysis of the coworker-to-future-boss arrangement. California Labor Code §2870 carve-out + MIT OSS is the cleanest structure; retainer, rev-share, gift-then-host, and "we'll figure it out" all have named failure modes. Drove the entire compensation recommendation and Round 3 C's ops model (Ramesh self-hosts so that OSS framing remains true).
